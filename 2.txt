Below is a step-by-step verification strategy that dovetails with Phase 3 (Java Client Integration) and gives us high confidence that the Rust Execution Module behaves exactly like the current Java EVM logic.

────────────────────────────────
1. Restate the Problem
────────────────────────────────
We are about to wire java-tron to a new Rust Execution Module. Before we allow it to produce canonical blocks, we must prove that:

• For every transaction, the post-state (balances, storage slots, code, nonces, logs, internal transfers, etc.) is bit-for-bit identical to what the existing Java implementation would have produced.  
• The system remains deterministic across OS/CPU builds.  
• Performance regressions are understood and acceptable.

java-tron lacks a built-in state-root Merkle tree, so we need an alternative equivalence oracle.

────────────────────────────────
2. Verification Approaches
────────────────────────────────
I propose running several complementary techniques instead of relying on a single signal:

A. Shadow (Dual) Execution with State-Diff Comparison  
   • Execute every block twice:  
     1. Java EVM (current path: `FullNode → TronVM`)  
     2. Rust EVM (new `ExecutionGrpcClient`)  
   • Capture each module’s “state digest” and compare.

B. Transaction-Level Golden Vectors  
   • Curate a suite of deterministic test cases (ERC-20 transfer, contract creation, self-destruct, tricky precompile calls, etc.).  
   • Run them in isolation and diff results.

C. Historical Chain Replay  
   • Re-execute e.g. the last 100 k mainnet blocks off-line, diff every N blocks, surface first mismatch.

D. Property-Based / Fuzz Testing  
   • Use a fuzzer (e.g.  `proptest` in Rust) to generate random but valid EVM TX sequences, run both engines, assert equivalence.

E. Canary Node in Testnet  
   • Attach a full-sync node that *verifies* blocks with Rust but *gossips* Java results; raise alerts on divergence.

────────────────────────────────
3. Designing a “State Digest”
────────────────────────────────
Because there’s no Merkle root, we compute our own compact hash:

1. After every transaction (or block), iterate over modified accounts.  
2. For each account concatenate: address || balance || nonce || codeHash || sorted(storageSlots).  
3. SHA-256 the concatenated bytes per account; XOR-reduce (or hash-chain) over all touched accounts into a single 32-byte digest.

Pros:  
• Works on RocksDB and in-memory maps.  
• Only needs modified keys → fast.  
• Deterministic across platforms.

Cons:  
• Still O(k) in number of modified keys per block, but k is small relative to full state.

We’ll build a small shared utility in `common/` (usable from both JVM via JNI or via gRPC) that receives a list of `AccountSnapshot` objects and returns the digest.

5. Step-by-Step Execution Plan
────────────────────────────────
Phase 3.1 SPI & Client Wiring  
• Code the `ExecutionGrpcClient` (PickFirst LB provider, deadline, retries).  
• Introduce `ExecutionMode` env var (default = EMBEDDED).  
• `SHADOW` mode → call both engines sequentially, capture in-memory `StateSnapshot`.

Phase 3.2 Digest Library  
• Rust function `fn digest(accounts: Vec<AccountSnapshot>) -> [u8;32]`.  
• Expose via gRPC **and** JNI for Java direct calls.  
• Unit-test cross-language determinism.

Phase 3.3 Shadow Harness  
• Where the block loop commits DB writes, insert:  
  ```java
  Digest dJava = DigestUtil.digest(snapshotJava);
  Digest dRust = DigestUtil.digest(snapshotRust);
  Preconditions.checkState(dJava.equals(dRust), "State mismatch at block %d", height);
  ```  
• Add configurable “panic‐or‐log” flag.

Phase 3.4 Golden Vectors  
• Pull fixture blocks from regression repo or craft them with Truffle tests.  
• Store expected receipts + digests in JSON; assert equality.

Phase 3.5 Historical Replay  
• Write Rust CLI `backend-replay` (or adapt existing `tron-cli`) to pipe blocks into both engines off-line.  
• Dump first mismatch for triage (tx index, differing key, expected vs actual value).

Phase 3.6 CI & Canary  
• GitHub Action: `./gradlew verifyShadow` (10 quick blocks).  
• Deploy canary node on private testnet; Prometheus metric `state_mismatch_total` triggers alert.

────────────────────────────────
6. Pros & Cons of Dual Execution
────────────────────────────────
Pros  
• Simple mental model: “old vs new” → hash match = good.  
• No need for Merkle tries or deep db hooks.  
• Can gradually phase out once confidence is high.

Cons  
• 2× compute per block in shadow mode (acceptable for verification, not production).  
• Requires deterministic snapshot order; we will sort keys to avoid nondeterminism.  
• Digest tool must be 100 % correct; we’ll cross-validate with full RocksDB dump hashes for spot-checks.

Mitigations  
• Shadow mode is compile-time/flag-gated; main-net production nodes will run single engine.  
• Provide sampling mode (e.g., 1 block per 1000) for long-running verification without full overhead.

7. Open Questions
────────────────────────────────
1. Coverage depth: Is replaying N blocks of mainnet sufficient, or do we need full historical sync?  
2. Storage-adapter snapshots: We need a cheap way to capture “modified account list” from both engines—Java side already tracks touched accounts; confirm availability on Rust side.  
3. Receipt equality: Should logs & gas used also be part of equivalence oracle? (recommended: yes).  
4. Timeline: Is one iteration of “no mismatch over 1 week canary” sufficient to cut over?
